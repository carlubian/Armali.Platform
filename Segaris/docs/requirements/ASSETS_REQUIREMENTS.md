# Assets Requirements

## Status

Phase 2 functional definition is complete. This document is the functional
source of truth for the initial Assets implementation plan.

## Purpose

Assets manages the durable physical objects the household owns and identifies
individually, where stock does not apply: furniture, appliances, electronics,
vehicles, tools, and similar items.

The initial module is intentionally simple. It is an enriched register of
individual assets: each asset carries a category, a location, a status, optional
identification details, an optional acquisition date, an optional expected end of
life, free notes, and photos. It does not track stock, monetary value, cost, or
maintenance history, and it is independent from every other business module.

## Initial Scope

- Manage individually identified durable objects as a single `Asset` entity.
- Describe each asset with a name, a category, a location, a status, an optional
  manual code, optional brand/model and serial number, an optional acquisition
  date, an optional expected end of life, optional notes, and attachments.
- Organise assets through Assets-owned `AssetCategory` and `AssetLocation`
  catalogues presented through Configuration.
- Present assets as a server-paginated table with a thumbnail column, search,
  filters, sorting, and a URL-aware popup editor.
- Surface launcher attention for assets approaching their expected end of life.
- Keep Assets independent from Capex, Opex, Inventory, Travel, Clothes, and other
  business modules in the initial release.

## Excluded Scope

The initial Assets implementation excludes:

- Stock, quantities, or any consumable behaviour (that is Inventory's domain).
- Monetary value, purchase price, cost, or any link to Capex.
- Maintenance, repair, or service history (that is Maintenance's future domain).
- Depreciation, amortisation, or financial reporting.
- Warranty as a first-class concept with provider, claims, or documents; only a
  single optional "expected end of life" date is modelled.
- Multiple locations per asset, location history, or movement tracking.
- Cross-module links to Capex, Opex, Inventory, Travel, Clothes, Archive, or
  Maintenance.
- Spanish translations.

## Asset Model

An asset contains at least:

- A required name.
- A required `AssetCategory`.
- A required `AssetLocation`.
- A status.
- An optional manual code, unique when present.
- An optional brand/model (free text).
- An optional serial number (free text).
- An optional acquisition date (civil date).
- An optional expected end of life (civil date).
- Optional notes.
- Attachments, optionally with one marked as the primary image.
- Visibility.

The initial model carries no monetary value, no quantity, and no reference to any
other module.

## Asset Status

Every asset has one of these fixed statuses:

- `Active`
- `Stored`
- `Retired`

The status is descriptive. It does not block editing, location or visibility
changes, or deletion by itself, and it is not managed through Configuration. It
exists to distinguish assets currently in use (`Active`), assets kept but not in
active use (`Stored`), and assets the household has decommissioned without
deleting (`Retired`), and to support filtering.

Only non-`Retired` assets participate in launcher attention.

## Identification

Assets carry optional identification fields to tell similar objects apart:

- **Code**: an optional, manually entered identifier. It is the household's own
  inventory reference and is not generated by the system. When present it must be
  unique across all assets (case-insensitive); when omitted the asset is
  identified by its name only.
- **Brand/Model**: optional free text.
- **Serial number**: optional free text.

None of these fields is required, and none participates in stock or financial
logic. The initial model does not model monetary value.

## Dates And Expected End Of Life

Assets store two optional civil dates:

- `AcquisitionDate`: when the asset entered the household.
- `ExpectedEndOfLifeDate`: when the asset is expected to reach the end of its
  useful life.

Both are optional and have no artificial boundary. The user-facing label for
`ExpectedEndOfLifeDate` is **"Expected end of life"**; the term "warranty" is not
used in the interface even though the date may, in practice, coincide with a
warranty expiry.

`ExpectedEndOfLifeDate` is the only date that drives launcher attention.

## Categories And Locations

Assets owns two module-specific catalogues:

- `AssetCategory`
- `AssetLocation`

Both are presented through Configuration and follow the established module-owned
catalogue behaviour:

- Administrator CRUD.
- Explicit ordering.
- Deletion-impact checks.
- Atomic replacement before deleting a referenced value.
- Privacy-neutral impact reporting.

Because both a category and a location are required on every asset, a referenced
value of either catalogue may only be **replaced**, never cleared, following the
Inventory category and location pattern. Replacement re-points the affected
assets to the target value.

### Initial Categories

The initial ordered category values are:

- `Furniture`
- `Appliances`
- `Electronics`
- `Vehicles`
- `Tools`
- `Other`

### Initial Locations

The initial ordered location values are:

- `Living room`
- `Bedroom`
- `Kitchen`
- `Office`
- `Garage`
- `Storage room`
- `Outdoors`
- `Other`

The one-time initialization behaviour matches the established Configuration,
Opex, Inventory, and Clothes catalogue pattern: values are initialized once and
are not reimposed after administrative changes.

## Attachments And Primary Image

- Assets may contain multiple attachments using the shared platform attachment
  policies and authorization model.
- An asset may mark one of its image attachments as the **primary image**. The
  table thumbnail uses the primary image; when none is marked, it falls back to
  the first image attachment; when there is no image, it shows a neutral
  placeholder.
- Any user who may access the asset may view, add, remove, and mark attachments
  as primary.
- Attachments inherit the visibility and authorization of their owning asset.

## Visibility And Authorization

Every asset uses the platform-standard visibility values:

- `Public`
- `Private`

New assets default to `Public`.

These rules apply:

- A user can view and edit their own assets and public assets.
- A private asset remains creator-only, including from administrators.
- Public collaboration follows the standard Segaris rule: any authenticated user
  may edit a public record.

These constraints are enforced by the backend regardless of the client.

## Deletion

Deletion is physical, immediate, and irreversible. An asset can always be
deleted; no other module references assets, so there is no deletion guard.
Deleting an asset removes its owned attachments.

## Validation

- Name is required, trimmed, not whitespace-only, and at most 200 characters.
- Category and location references are required and valid.
- Status and visibility are known values.
- Code is optional and at most 50 characters; when present it is trimmed, not
  whitespace-only, and unique across all assets (case-insensitive).
- Brand/model is optional and at most 200 characters.
- Serial number is optional and at most 200 characters.
- `AcquisitionDate` and `ExpectedEndOfLifeDate` are optional civil dates with no
  artificial boundary.
- Notes are optional and at most 4,000 characters.

## Creation Defaults

A new asset starts with:

- Status `Active`.
- Visibility `Public`.
- The first available category by `SortOrder`, then `Id`.
- The first available location by `SortOrder`, then `Id`.
- No code.
- No brand/model and no serial number.
- No acquisition date and no expected end of life.
- No notes.
- No attachments and no primary image.

## Module Entry And Navigation

Opening Assets takes the user directly to the assets table. Assets does not have
an initial overview or dashboard.

Creating, viewing, and editing assets happens in a popup dialog over the table,
following the established Segaris URL-aware dialog pattern, so table state
survives dialog open and close without a reload.

## Assets View

The primary view is a server-paginated table. It includes at least these
columns:

- Thumbnail (primary image, first image, or placeholder).
- Name.
- Code.
- Category.
- Location.
- Status.
- Expected end of life.
- Visibility.

The default ordering is name ascending, then identifier ascending.

The table supports:

- Partial search across name, code, brand/model, serial number, and notes.
- Exact filters for category, location, status, visibility, and creator.
- User-controlled sorting and bounded pagination following platform conventions.

Search, key filters, sort, page, and page size should be URL-backed where
practical.

## Attention

The Assets launcher card requires attention when at least one accessible asset
satisfies both conditions:

- Status is not `Retired`.
- `ExpectedEndOfLifeDate` is set and falls within the next 30 days in
  `Europe/Madrid`, that is, within the inclusive window from today to today plus
  30 natural days.

Assets whose expected end of life is already in the past do not, by themselves,
activate attention; only the upcoming window counts. Only accessible assets count
for the current user.

The launcher exposes only the platform-standard boolean attention state.

## Acceptance Criteria

The initial Assets definition is satisfied when:

1. Authenticated users can create, query, edit, and irreversibly delete visible
   assets with the documented fields, defaults, validation, and privacy rules.
2. An asset is a single individually identified entity with no stock, monetary
   value, or cross-module references.
3. Asset statuses `Active`, `Stored`, and `Retired` are available and
   descriptive, blocking no operation by themselves.
4. The optional code is unique across all assets when present, and brand/model
   and serial number are optional free-text identification fields.
5. Acquisition date and expected end of life are optional civil dates, and the
   expected-end-of-life field is surfaced to the user as "Expected end of life".
6. Public collaboration and private isolation follow the Segaris visibility
   baseline for assets.
7. Assets support multiple attachments with an optional primary image that drives
   the table thumbnail, falling back to the first image and then a placeholder.
8. Assets-owned `AssetCategory` and `AssetLocation` are initialized once and
   managed through Configuration with CRUD and reorder.
9. Deleting a referenced category or location requires replacement, never
   clearing, atomically and without disclosing private assets.
10. The assets table presents the documented columns with the documented search,
    filters, sorting, and bounded pagination, and the editor is a URL-aware popup
    that preserves table state.
11. Assets attention is true exactly when the current user can access at least one
    non-`Retired` asset whose expected end of life falls within the next 30 days
    in `Europe/Madrid`.
12. SQLite and PostgreSQL migrations, backend unit/integration/architecture
    tests, frontend component tests, and a representative Playwright journey
    verify the supported behaviour and privacy boundaries.

## Deferred Decisions

- Whether assets should later gain monetary value, purchase, or cost attributes,
  or an optional link to the Capex entry that recorded their purchase.
- Whether the future Maintenance module should reference assets and record
  repairs or service history against them.
- Whether warranty should become a first-class concept with provider, documents,
  and claims, beyond the single expected-end-of-life date.
- Whether assets should support multiple or historical locations.
- Whether the code should later support optional system-assisted generation.
- Whether future Analytics will consume Assets read contracts.
